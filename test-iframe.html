Sandbox Security Considerations

Permissions ALLOWED (minimal set required for functionality):

  • allow-scripts: Required for the JavaScript animation to execute
  • allow-same-origin: Required for:
    • Loading the iframe's own CSS and JavaScript files
    • Accessing external resources (Google Fonts, background images) via CORS
    • Proper functioning of the animation

Permissions BLOCKED (not needed, remain restricted):

  • ‚úÖ allow-forms: Blocked - No forms are submitted in this application
  • ‚úÖ allow-popups: Blocked - No popup windows are created
  • ‚úÖ allow-top-navigation: Blocked - No navigation to other pages
  • ‚úÖ allow-modals: Blocked - Uses in-page elements only (no alert/confirm/prompt dialogs)
  • ‚úÖ allow-downloads: Blocked - No download functionality
  • ‚úÖ allow-orientation-lock: Blocked - Not needed
  • ‚úÖ allow-pointer-lock: Blocked - Not needed
  • ‚úÖ allow-presentation: Blocked - Not needed

Security tradeoffs:

  • ‚úÖ Good: Only 2 permissions granted (scripts + same-origin), all other sandbox-controlled permissions blocked
  • ‚úÖ Good: No form submission, popups, navigation, or modal dialogs possible
  • ‚úÖ Good: This is the absolute minimum permission set for the animation to work
  • ‚ö†Ô∏è Consideration: The combination of allow-scripts + allow-same-origin removes some sandbox protections
  • ‚ö†Ô∏è Consideration: If iframe content were compromised, it could access cookies and localStorage

Troubleshooting

  • GitHub Pages iframe blocked: Expected. Deploy to Netlify/Vercel to fix.
  • Console shows CSP errors: Check that headers are properly configured.
  • Blank iframe: Check browser console (F12) for specific error messages.
  • Resources not loading: Verify HTTPS and CORS settings.

Header Requirements

For iframe embedding to work, the server should send:

Content-Security-Policy: frame-ancestors *

And should NOT send:
X-Frame-Options: DENY

‚úÖ The correct CSP header is automatically configured in the deployment files (_headers, netlify.toml, vercel.json)
